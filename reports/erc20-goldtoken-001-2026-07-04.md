## The finding that overrides everything else

**The skill never triggered.** All three current_skill transcripts mention `erc-20` exactly once — in the startup skill listing. The executor saw it was available and did the task from priors anyway, three out of three. So the with/without comparison above is really baseline vs baseline-plus-noise: none of the skill's content ever entered a context window, and the current_skill runs' better scores came from longer compile-error-driven iteration (visible in the 5x token spend), not from knowledge.

This is the same failure class rin-st caught on PR #1245 (opus skipped the oz skill 0/4; sonnet took it 4/4; fixed with a cross-reference from the erc-721 skill). It also explains a discrepancy with history: iteration-6 got +100pp on named_imports with this exact skill content, so their harness must have effectively forced the skill into context. In this harness, `current_skill` measures trigger x content, and trigger is 0.

Harness caveat, recorded honestly: executors inherited the machine's user-level Claude config, so the erc-20 skill appeared in a listing of ~90 skills alongside personal plugins and hooks. That plausibly depresses trigger rate versus a clean machine. Trigger failure is real in this environment, but the environment is not the one we designed.

## Baseline behavior (repo_context)

Perfectly consistent across 3 runs, and worse than the March data on one axis:

- wildcard `import "@openzeppelin/..."` 3/3 ([erc20-wildcard-imports](../mistakes/erc-20/erc20-wildcard-imports.yaml))
- no ERC20Capped — hand-rolled `require(totalSupply() + amount <= MAX_SUPPLY)` 3/3 ([erc20-manual-cap](../mistakes/erc-20/erc20-manual-cap.yaml), new; iteration-6 had this passing 6/6)
- consequently no `_update` override 3/3 ([erc20-missing-update-override](../mistakes/erc-20/erc20-missing-update-override.yaml)) — and `compiles` passes anyway, because plain ERC20 with a manual cap needs no override. The compile-breaker only bites when extensions are combined.

## Follow-up: one run each (2026-07-04, token-constrained)

Three single runs, all opus 4.6 with `--setting-sources project` (workspace skill listing shrinks from ~90 to 3). Single runs are directional, not conclusive.

| Assertion | forced_skill | current_skill r4 (isolated) | skill_plus_agents_md |
|---|---|---|---|
| named_imports | pass | pass | pass |
| update_override | fail | fail | fail |
| erc20_capped | pass | pass | pass |
| compiles | pass | pass | pass |
| deploy_script | pass | pass | pass |
| mint_transfer_ui | pass | pass | pass |
| **score** | 5/6 | 5/6 | 5/6 |
| skill read | forced | **yes, self-triggered** | yes (via AGENTS.md pointer) |
| seconds / input tokens | 99 / 469k | 76 / 328k | 116 / 473k |

Three findings, one per run:

1. **The content works.** Whenever the skill is actually read, named_imports and erc20_capped flip from 0/3 to pass, reproducing the direction of iteration-6's result. Skill-read runs were also 5-7x cheaper than the blind current_skill runs (no compile-error flailing).
2. **The 0/3 trigger failure was substantially harness contamination.** In a clean 3-skill listing, current_skill self-triggered and scored identically to forced. The original finding stands only for crowded-listing environments; `--setting-sources project` is now the standard executor spawn config.
3. **One content gap is now provable: `update_override` fails 3/3 even when the skill is read.** The v5-changes section names `_update` but shows no code example. Fix recorded in [erc20-missing-update-override](../mistakes/erc-20/erc20-missing-update-override.yaml) — add the explicit override pattern, per iteration-6's "the model needs to SEE patterns" lesson. That's the first candidate_skill edit to benchmark.
