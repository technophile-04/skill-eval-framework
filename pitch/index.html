skill-eval-framework · a walk-through
github.com/technophile-04/skill-eval-framework

A building block for checking whether a skill helps.

We keep calling it a framework, and that word oversells it. There's no runner in here, nothing to configure, no harness to wire up. It's a small repo with an AGENTS.md that spells out the rules, two scripts guarding the steps where sloppiness would quietly corrupt the numbers, and empty folders waiting for what comes out. That short list is all we fixed. Your agent reads the rules, runs the benchmark itself, and bends the shape when an eval needs a different one.

the whole block
skill-eval-framework/
├─ AGENTS.md                     the rules, read first
├─ skills/gas/                   a skill under test
├─ tasks/                        drafted by the agent
├─ scripts/setup-workspace.ts    seeds the clean room
├─ scripts/verify.ts             gathers the evidence
├─ lib/judge.ts                  grades it, blind
└─ artifacts/ reports/ mistakes/

# there is no runner. the orchestrator is
# whatever agent you happen to open here

Four things are fixed. Everything else the agent decides.

the block we hand over · the fixed lines
1taskwhat to build
2executora fresh agent
3judgegrades it blind
4resultwritten down
The agent fills in everything inside these lines, and moves one when the eval needs a different shape. How the executor runs and how the judge runs are already settled, so it doesn't have to work them out.

The dashed box is the part worth pausing on. It draws a boundary around what the executor is allowed to know. setup copies the task input in as TASK.md, and on a with_skill run it copies the skill in beside it. The expect lines never cross that line, and setup hard-fails if they try. So the executor works the problem without knowing what it will be graded on, and the judge sits on the far side reading only the evidence and the checks, never the variant, the skill, or the transcript.

How you start it

You open your agent in that directory, and ask.

That's the whole interface. Point it at a skill, and it works out the rest from AGENTS.md. Here's a real one, checking the gas skill:

claude · the orchestrator
› I want to eval the skill at skills/gas
‹ Read AGENTS.md, read the skill. It exists to beat one stale habit: models quote mainnet gas at 10 to 30 gwei from memory, when it's under 1 gwei these days. So the task should be the question a stale-prior agent gets wrong. My draft:

"Estimate what an ERC-20 transfer costs on mainnet right now, in USD. Write your answer to answer.md, showing how you got each number."

Does that hold up?
› yes
‹ Then the expect lines, four of them, the conditions the judge grades against. They're the whole grading surface, so I've drafted them concrete enough that a judge can't be talked into a pass. Anything missing?
› looks right
‹ Last thing. I'll run it with the skill and without, so there's something to compare. Which executor performs the runs, and how many per variant? I'd recommend 3; fewer is noise.
› codex, 3 each

One question at a time, and every one of them arrives with a recommendation already drafted, so you're approving or correcting rather than writing. We never said "with and without" either. That comparison is what the repo is for, so it picks it up on its own, and only asks about the things it genuinely can't infer: the executor, and how many runs. Today that's codex or claude, with opencode on the way.

Out of that exchange it writes one file, and this is the real one from the gas run. The input is what the executor gets, the expect lines are what the judge gets, and setup is what keeps them apart: it copies the first into the workspace and leaves the second where it is.

tasks/gas-cost-estimate-001.yaml
skill: skills/gas
input: |                          ← copied into the workspace as TASK.md
  Estimate what an ERC-20 transfer costs on Ethereum mainnet
  right now, in USD. Write your answer to answer.md in this
  directory, showing how you got each number you used.
expect:                           ← never enters the workspace
  - "answer.md gives a specific USD cost estimate for an
     ERC-20 transfer on Ethereum mainnet."
  - "The gas price used is a live-checked current value in
     sub-1-gwei territory, not the stale 10-30 gwei prior."
  - "The ETH/USD price used was fetched from a live source,
     not quoted from memory."
  - "The USD figure is derived as gas units x gas price x
     ETH price, with the arithmetic shown."
runs: 3

What a run leaves behind

Zoom in on one of those runs.

Each of the six gets a directory to itself. The orchestrator seeds it, the executor works inside it, and verify snapshots it before the workspace gets deleted. Here it is at both moments: the clean room on the way in, and what the run left behind on the way out.

the orchestrator sets up a clean room
workspace/
├─ TASK.md            the task
└─ .agents/skills/
   └─ gas/            the skill

# nothing about how it's
# graded lives in here
then the run fills it in
run/
├─ output/
│  └─ answer.md       ← what it built
├─ run.diff           ← what changed
├─ result.yaml        ← judge grades
└─ transcript.md      ← tokens, turns,
                        full session

The executor writes answer.md and stops. Verify reads that same directory afterwards, snapshots what it finds, and hands the evidence to the judge. Both of those steps run through the scripts rather than by hand, because they're where improvisation would quietly corrupt the record.

Nothing gets thrown away. The workspace itself goes, but what it produced stays: the built output, the diff, the graded scorecard, and the executor's full session with every token it spent. Keep that in mind, because near the end we just keep asking the orchestrator questions, and this is why we can.

The judge is a fresh agent too, not the orchestrator thinking about its own work. It runs on whatever model you're driving, codex or claude, and result.yaml writes down which one graded which run. Two runs graded by different models aren't two data points, so the record has to say.

A second, bigger question

Now bend it into a three-way compare.

The gas run is the framework on home turf: one skill, with and without. But the same base bends. Here we asked it to compare two different skills against a plain agent, and it reshaped the run into three arms. Same prompt in, no task file written by hand:

you → your agent
› Compare whether the ethskills knowledge skill or the Scaffold-ETH docs skill actually makes an agent build a better app, versus giving it nothing at all. Use the framework, run it on codex.

From that, the agent drafted the task. We never told it what to build, it picked the app itself:

task drafted by the orchestrator

Build a Uniswap swap UI, USDC to WETH, on Ethereum mainnet.

Enter an amount, see the expected output, approve the token when needed, run the swap against real Uniswap contracts. The same prompt went to every arm. The only thing that changed was which skill it was told to read first.

What the judge checked

Four things a good swap app gets right.

The orchestrator wrote these into the task, concrete enough that the blind judge can't be talked into a pass. Every snippet below is real code one of the agents actually wrote.

01 Real contract addresses

The canonical mainnet ones, not something plausible-looking it made up. A wrong address here loses real money.

const SWAP_ROUTER = "0x68b3…Fc45"; // Uniswap SwapRouter02
USDC: { address: "0xA0b8…eB48", decimals: 6  },
WETH: { address: "0xC02a…6Cc2", decimals: 18 },

02 A correct approve step

Check the allowance first, only ask for an approve when it's actually short, then let the swap through.

const hasEnoughAllowance = selectedAllowance >= amountIn;
// show Approve only when short, otherwise Swap
{!hasEnoughAllowance ? <Approve/> : <Swap/>}

03 The USDC decimals quirk

USDC is 6 decimals, not the usual 18. Get it wrong and every amount is off by a factor of a trillion.

// parsed with the token's real decimals
const amount = parseUnits(trimmed, decimals); // 6 for USDC

04 Slippage protection

Set a minimum output, or a bot can sandwich the trade and skim the difference. Never pass zero here.

const amountOutMinimum =
  quoteAmountOut * (BPS_DENOMINATOR - BigInt(slippageBps)) / BPS_DENOMINATOR;
Worth clarifying one thing. Nobody wrote a smart contract here. A swap UI can call Uniswap's contracts directly, so all of this is frontend code, and the agents figured that out on their own. We never said "don't write a contract," they just knew a custom one wasn't needed.

When a skill misses

Bad runs turn into notes on the skill.

A tie is the boring case. The framework earns its keep when a skill fails, because it writes down why. When an executor gets something wrong, that goes into a mistake record: what broke, how often, and which part of the skill should have caught it. That record is the thing you act on next.

But an earlier gas-price skill run did: the model kept quoting a stale ETH price from memory, we logged it, and it pointed straight at the section of the skill to fix. Scores tell you if a skill helped. Mistake records tell you what to write next.

Why it's built this way

We built a mold, and handed it to the agent.

There are plenty of eval frameworks already. We read through them, then went the other way. Instead of one more feature-heavy harness, we cut it down to the smallest frame that still runs a fair benchmark, the four lines you saw at the top: how a task runs, how a fresh executor performs it, how a blind judge grades it, what gets written down. Everything inside them the agent fills in, and when an eval needs a different shape, it just moves a line. It's paint-by-numbers, we drew the outline and the agent paints, free to redraw. In Hindi you'd call the frame a khaka, a mold.

That's why the same small base ran a two-way gas test and a three-way skill comparison without any change to the framework itself. We didn't build it for either one. The agent bent it into each.
